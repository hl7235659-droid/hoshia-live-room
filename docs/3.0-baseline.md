## 已实现功能

### 1. 私密直播间入口

用户需要先通过房间口令，再用一次性注册码注册或用账号登录，才能进入直播间。

实现方式：

- 网关提供 `/api/auth/gate`、`/api/auth/register`、`/api/auth/login`
- 登录态通过 HttpOnly session cookie 保存
- 用户和注册码存储在 SQLite
- 前端在进入主界面前先做 gate 和登录检查

关键文件：

- `gateway/src/account-routes.js`
- `gateway/src/security.js`
- `gateway/src/database.js`
- `frontend/src/main.tsx`

### 2. 实时直播间

房间内的在线状态、消息、AI 回复、音乐状态、Hoshia 状态都通过 WebSocket 实时同步。

实现方式：

- `/ws/live` 负责连接建立和鉴权
- 连接后下发房间状态、音乐状态、Hoshia 状态
- 弹幕消息进入后端统一处理，再广播给前端
- heartbeat 负责 ping / terminate，避免僵尸连接

关键文件：

- `gateway/src/live-room-websocket.js`
- `gateway/src/server.js`
- `frontend/src/main.tsx`

### 3. 账号与资料

用户可以修改昵称、头像、弹幕颜色，也可以修改密码和完成 onboarding。

实现方式：

- 后端负责资料校验、数据库更新、session 刷新
- 前端把账号设置拆成独立面板，减少主界面耦合

关键文件：

- `gateway/src/account-routes.js`
- `frontend/src/components/AccountPanels.tsx`

### 4. Hoshia 回复系统

Hoshia 可以根据房间上下文、用户消息和模块状态生成回复。默认可以用 mock 模式，也可以切到 AstrBot bridge。

实现方式：

- `AI_MODE=mock` 时本地生成模拟回复
- `AI_MODE=astrbot` 时通过桥接插件调用外部回复能力
- 回复前会组装人格 prompt、最近消息、现实上下文和模块上下文
- 支持 pending / streaming 风格前端展示

关键文件：

- `gateway/src/ai-adapter.js`
- `gateway/src/message-router.js`
- `gateway/src/hoshia-persona.js`
- `astrbot_plugin_live_room_bridge/`

### 5. 模块化上下文与记忆

项目已经不再靠单个主 prompt 硬塞所有背景，而是通过模块上下文、模块事件和记忆提纯来让 Hoshia 理解房间状态。

实现方式：

- `module_context` 给 Hoshia 提供当前模块能力和状态
- `module_events` 记录最近的用户行为归因
- `module_memory_events` 只保留适合提纯成长期记忆的事件
- 新模块优先做 provider，不直接反复修改主人格 prompt

关键文件：

- `gateway/src/module-context.js`
- `gateway/src/hoshia-life-memory.js`
- `gateway/src/hoshia-daily-canon.js`

### 6. 音乐房间实验

已经有点歌、队列、当前播放、控制状态和音乐模块上下文。

实现方式：

- `MusicService` 管理播放状态和队列
- 前端把音乐面板单独拆出
- Hoshia 可以读取当前播放、待播队列和点歌人信息

关键文件：

- `gateway/src/music-service.js`
- `gateway/src/module-context.js`
- `frontend/src/components/MusicPanels.tsx`

### 7. 时间线 / 动态 / 评论

Hoshia 可以自动发动态，用户可以点赞和评论，Hoshia 还能延迟回复评论。

实现方式：

- 后端按定时或手动生成 posts
- 评论回复服务按 tick 运行，不阻塞主消息流
- 前端把时间线覆盖层拆开，主页面更干净

关键文件：

- `gateway/src/hoshia-daily-post.js`
- `gateway/src/hoshia-comment-reply.js`
- `gateway/src/hoshia-life-memory.js`
- `frontend/src/components/TimelineOverlay.tsx`

### 8. 视觉状态与 Live2D fallback

Hoshia 的外观状态会跟随 mood、activity、energy、social_need 和当前 PNG 变化。Live2D 可配置，但默认稳定基线仍是 PNG fallback。

实现方式：

- 后端维护公开视觉状态
- 前端优先尝试 Live2D
- 没有模型或加载失败时自动回退到 PNG

关键文件：

- `gateway/src/hoshia-visual-state.js`
- `frontend/src/CharacterStage.tsx`

## 当前边界

- 真实 Live2D 不是默认强制接入，仍以 PNG fallback 作为稳定基线
- TTS、礼物、动作事件还属于后续演进方向
- `/api/hoshia/*` 还保留在主协调文件里，后续如果继续拆，只适合分批做纯抽取
- 本阶段默认不再做大重构，只在现有模块化机制上继续加功能

## 3.0 接手规则

- 新模块优先走 `module_context` / `module_events` / `module_memory_events`
- 不要为了单个功能反复改 Hoshia 主 prompt
- 不提交 `.env`、数据库、日志、`dist`、`node_modules`、缓存、密钥
- 部署前必须备份 staging 目录，并保留服务器 `.env`、`gateway/data` 和 tunnel 配置
- 任何新改动都优先保持可回滚、可单独验证
